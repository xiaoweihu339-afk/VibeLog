# VibeLog v0.1 Design

## Purpose

VibeLog is a Markdown-first standard for recording how a vibe product idea grows from a messy thought into a clearer idea, an implementation process, and eventually a shareable product.

The standard must work before the website exists. A user or agent should be able to create and maintain a `vibe-log.md` locally, then export a `vibe-log.json` that a future website can upload into a private idea library, public idea space, progress page, or product showcase.

## Core Principles

- Human-readable first: the Markdown file should feel like a clear creative record, not a database dump.
- Agent-readable by structure: fixed headings, YAML frontmatter, and predictable fields let agents read, update, and hand off work reliably.
- Markdown is the source of truth: JSON is generated from Markdown, not edited manually by default.
- One-line vibe first: every log starts with one sentence explaining what is being vibed.
- Preserve evolution: old ideas are not deleted when the direction changes; changes are recorded in `Idea Evolution`.
- Support handoff: a new agent should be able to read the log and understand what exists, what is done, what is pending, and what to do next.
- Mid-project capture: the log can be created or updated in the middle of a vibe product, including reconstructed prior context.
- Prompt traceability: engineering execution prompts used for vibecoding must be recorded separately from general idea notes, preferably with exact prompt text.
- Conversation distillation: chat-like idea exploration should not be copied as transcript; extract only idea changes, human judgment, decisions, and open questions.
- Development continuity: normal project development work, including features, bug fixes, refactors, tests, docs, releases, and chores, can be recorded in the same log.
- Human-in-the-loop clarity: human judgment points should be recorded explicitly, including direction, taste, tradeoffs, approval, rejection, and risk decisions.
- Public content is intentional: the user controls whether the idea, progress, or product is private or public.

## Files

Each vibe project can contain:

```txt
vibe-log.md
vibe-log.json
```

`vibe-log.md` is the primary file. `vibe-log.json` is derived from it for upload and integration.

# VibeLog

## One-Line Vibe

A one-sentence description of what I am vibing.

## Current Idea

The current version of the idea. This section can be updated as the user's thinking changes.

## Idea Expansion

Expand the one-line idea:
- What problem it solves
- Who it is for
- Why it matters
- Possible core features
- Possible use cases

## Idea Evolution

Record idea changes without deleting history.

### YYYY-MM-DD HH:mm
**Type:** initial / expansion / pivot / refinement / removal

**Before:** The previous idea

**After:** The new idea

**Reason:** Why the change happened

## Decisions

Record decisions that have been made.

### YYYY-MM-DD HH:mm
**Decision:** What decision was made

**Why:** Why this decision was made

## Human-in-the-Loop

Record where human judgment shaped the vibe product.

### YYYY-MM-DD HH:mm
**Type:** direction / scope / taste / tradeoff / approval / rejection / risk / naming / prioritization

**Human Input:** What the human decided, corrected, approved, rejected, or clarified

**Agent Proposal:** What the agent suggested, if relevant

**Final Decision:** What direction was chosen

**Why It Mattered:** Why this human decision changed the product

**Impact:** What changed in the idea, design, implementation, or next steps

## Open Questions

Questions that are not resolved yet.

## Implementation Status

### Current State
Summarize the current implementation state in natural language.

### Completed
- Completed item

### In Progress
- Item currently in progress

### Pending
- Pending item

### Blocked
- Blocked item and reason

### Next Actions
- Next smallest action

### Important Context for Next Agent
- Key files, key decisions, work not to repeat, and things to watch out for

## Project Context

### Repo / Workspace
Project location or repository link.

### Important Files
- `path/to/file`: purpose

### Run / Test Commands
- `command`

### Known Issues
- Known issue

### Do Not Change
- Areas or decisions that should not be changed for now

## Execution Prompts

Record engineering prompts used to execute vibecoding work. This section is a prompt ledger, not a chat transcript.

### YYYY-MM-DD HH:mm
**Agent / Tool:** Agent or tool used

**Prompt Type:** build / refactor / debug / test / design / deploy / research

**Prompt Visibility:** hidden / summary / full

**Recording Mode:** exact / redacted / reconstructed / summary_only

**Prompt Summary:** Safe summary of the execution prompt

**Prompt Text:** Exact prompt text for engineering execution prompts, unless redacted for secrets or privacy

**Result:** What the prompt produced or changed

**Reuse Notes:** How a future agent should reuse or avoid this prompt

Rules:

- If a prompt directly asks an agent to build, edit, debug, test, refactor, design implementation, deploy, inspect files, or run commands, record it in `Execution Prompts`.
- Record exact prompt text by default for engineering execution prompts.
- If exact text contains secrets, credentials, personal data, or sensitive proprietary content, redact only the sensitive parts and set `Recording Mode` to `redacted`.
- If reconstructing an old execution prompt from memory or history, set `Recording Mode` to `reconstructed`.
- Do not store ordinary idea-chat messages here unless they directly instruct execution. Extract idea-chat content into `Idea Evolution`, `Human-in-the-Loop`, `Decisions`, or `Open Questions`.

## Development Log

Record normal project development events. Use this for engineering-level work that is more specific than product progress.

### YYYY-MM-DD HH:mm
**Type:** feature / bugfix / refactor / test / docs / chore / release / config

**Summary:** Short summary of the change

**Files Changed:** Important files changed

**Details:** What changed and why

**Bug Symptom:** For bugfix entries, what was broken

**Root Cause:** For bugfix entries, why it was broken

**Fix:** For bugfix entries, what fixed it

**Verification:** How the change was verified

**Follow-up:** Any remaining follow-up work

## Vibe Progress

Record actual progress.

### YYYY-MM-DD HH:mm
**Stage:** idea / brief / prototype / mvp / beta / shipped / paused / abandoned

**What Happened:** What moved forward in this update

**Tools Used:** Tools used

**Problems:** Problems encountered

**Next:** Next step

## Website Mapping

The website can use `visibility` to decide where a log appears:

- `private`: My Vibe Studio private library.
- `public_idea`: Public Idea Space.
- `public_progress`: public progress page or vibe branch.
- `public_product`: Product Showcase.

Public idea behavior:

- If a user publishes an idea to Public Idea Space, it means others may create their own vibe branches from that idea.
- The original idea author should remain credited.
- Each branch owner controls how much of their branch is public, including code and prompt visibility.

## Agent Maintenance Rules

- Always keep `One-Line Vibe` current and concise.
- When the user's direction changes, update `Current Idea` and append an `Idea Evolution` entry.
- Do not erase old idea history unless the user explicitly asks to redact it.
- Record concrete decisions in `Decisions`, separate from open questions.
- Record human judgment in `Human-in-the-Loop` when the user changes direction, sets taste, approves or rejects an option, chooses a tradeoff, limits scope, or makes a risk decision.
- Keep `Implementation Status` current enough that another agent can continue the work.
- Use `Vibe Progress` for chronological progress updates.
- Use `Execution Prompts` for prompts that were used to execute building, refactoring, debugging, testing, implementation design, deployment, file inspection, command execution, or research work. Record exact prompt text unless redaction is necessary.
- Do not copy general idea chat into `Execution Prompts`; distill it into idea evolution, human-in-the-loop records, decisions, or open questions.
- Use `Development Log` for normal project development events. For bug fixes, capture symptom, root cause, fix, and verification when known.
- If the log is created mid-project, reconstruct prior ideas and progress from the current conversation, local files, git history, and user-provided context; mark uncertain reconstruction clearly.
- Ask before changing any `visibility` from private to public.
- Treat `vibe-log.md` as the source of truth and regenerate JSON from it.

## First Project To Use It

This VibeLog standard should be used to record the evolution of the VibeLog product itself:

- Initial idea: a website where vibe coding people can share vibe-built works.
- Expanded idea: a platform where anyone can record ideas, track vibe progress, publish public ideas, invite remix through branches, and showcase mature products.
- Infrastructure idea: a VibeLog skill that records one-line ideas, idea expansion, idea evolution, implementation status, and progress in a standard format.
